01 — El Problema

El caos del inventario manual y las desconfiguraciones lógicas

En redes medianas y grandes, llevar el control de qué dispositivos están activos, qué versiones de firmware ejecutan y qué direcciones IP tienen asignadas mediante planillas de cálculo estáticas es una receta segura para el desastre operativo.

La falta de un sistema centralizado para dar de Alta, Baja o Modificar (ABM) equipos provoca inconsistencias críticas, como IPs duplicadas, configuraciones desactualizadas o routers huérfanos sin registrar. El desafío consistía en diseñar una aplicación lógica que sirviera como única fuente de verdad (Single Source of Truth) para el inventario, conectando el almacenamiento relacional con la posibilidad de auditar de forma directa el estado de los equipos.

Insight clave: Un error de transcripción manual en una dirección de gestión puede dejar fuera de línea a un segmento entero de la topología. Automatizar el ABM elimina el factor de error humano.

02 — Metodología

Ciclo de desarrollo estructurado para NetDevOps

Para garantizar la confiabilidad y la modularidad del sistema de gestión, el desarrollo se segmentó en etapas claras de arquitectura de software aplicada a la infraestructura:

  • 01
    Modelado del Esquema Relacional
    Diseño de las tablas de base de datos para almacenar hosts, familias de equipos, asignación de subredes e historiales lógicos de modificación.
  • 02
    Desarrollo de la Lógica en Python (CRUD)
    Programación de las funciones nucleares para el manejo de Altas (con validación estricta de formatos IP), Bajas lógicas (preservando logs) y Modificaciones.
  • 03
    Integración con Capa de Abstracción de Red
    Implementación de conectores programáticos para que, al dar de alta o modificar un equipo, el sistema valide la accesibilidad física del nodo mediante protocolos estándar de comunicación.
  • 04
    Pruebas unitarias de consistencia
    Ejecución de baterías de pruebas para asegurar que modificaciones concurrentes o ingresos inválidos de red sean rechazados correctamente por el backend.

La construcción del ABM resolvió el problema estructural de la gobernanza de datos sobre la infraestructura. La aplicación demostró que la programación estructurada en Python y el manejo correcto de bases de datos son herramientas indispensables para cualquier administrador moderno. Como evolución directa, el sistema quedó preparado para interactuar mediante Webhooks con las herramientas de monitoreo desplegadas previamente (como telemetría o análisis de flujos), cerrando el círculo de gestión automática de la red.
